# 微服务多版本请求路由:tw-1f1fb:

在微服务架构中，允许多版本的微服务共存是一种常见的做法。通过支持多版本共存，可以实现平滑的服务升级和迭代，同时满足不同客户或用户的需求。


jvs中使用基于请求头方式确定版本,并使用nacos元数据`jvs-rule-ua=2.1.8`确定版本号

并行开发和测试：使用多版本可以使团队成员在同一时间内独立进行不同版本的开发和测试工作。这样可以有效地提高开发者的并行工作能力，加快项目的迭代速度。

如: 开发人员(a,b)开发相同xx_ui,(c,d) 开发相同xx_service
当c,d开发时的模块存在各自的jvs-rule-ua版本时,a,b指定:tw-1f449:对应的版本即可每次都请求对应的服务

核心代码
~~~

    /**
     * 获取默认服务实例
     * <p>
     * 1、找是否有与网关相同的版本号,随机一个
     * 2、找是否有默认标识版本号
     * 3、随机找一个版本号
     *
     * @param instances 服务实例集合
     * @param version   版本号(路由标识)
     * @return 服务实例
     */
    private static ServiceInstance defaultVersionOrRandomInstance(DiscoveryClient discoveryClient, List<ServiceInstance> instances, String version) {
        List<ServiceInstance> matchedInstances = instances.stream()
                .filter(instance -> MapUtil.getStr(instance.getMetadata(), SysConstant.VERSION, "-").contains(version))
                .collect(Collectors.toList());
        if (ObjectUtils.isNotEmpty(matchedInstances)) {
            return matchedInstances.get(RandomUtil.randomInt(matchedInstances.size()));
        }
        //如果没有找到版本号，默认使用网关相同的版本号。
        ServiceInstance serviceInstance = discoveryClient.getInstances(SpringContextUtil.getApplicationContextName())
                .stream()
                .findFirst().get();
        //如果有版本号优先找和网关相同的版本
        String gatewayVersion = serviceInstance.getMetadata().getOrDefault(SysConstant.VERSION, SpringContextUtil.getVersion());
        List<ServiceInstance> serviceInstances = instances.stream()
                .filter(instance -> MapUtil.getStr(instance.getMetadata(), SysConstant.VERSION, "-").contains(gatewayVersion))
                .collect(Collectors.toList());
        if (ObjectUtils.isNotEmpty(serviceInstances)) {
            return serviceInstances.get(RandomUtil.randomInt(serviceInstances.size()));
        }
        // 找有没有默认版本号
        Optional<ServiceInstance> first = instances.stream().filter(e -> e.getMetadata().containsValue(SysConstant.DEFAULT)).findFirst();
        // 随机一个版本号
        return first.orElseGet(() -> instances.get(RandomUtil.randomInt(instances.size())));
    }

~~~

下面是一些常见的方法来实现微服务多版本共存：

1. 路由和网关：使用路由和网关层来根据请求的版本号或其他标识，将请求转发到对应版本的微服务。通过适当的规则配置和路由策略，可以实现不同版本的微服务共存。

2. 基于URL路径：在微服务的URL路径中添加版本号作为一部分，例如`/v1/service`和`/v2/service`。通过解析URL中的版本号，可以将请求路由到对应的微服务版本。

3. 基于请求头：在请求头中添加版本号作为自定义的头部信息，例如`X-API-Version`。微服务可以根据请求头中的版本号进行版本匹配和路由转发。

4. 基于服务发现和注册：使用服务发现和注册机制，如Consul、nacos、Eureka等，将不同版本的微服务注册到注册中心。客户端可以通过查询注册中心来获取和调用特定版本的微服务。

5. 负载均衡策略：在负载均衡策略中添加更多的权重或规则，以达到调节不同版本微服务的流量分配比例的目的。可以根据版本号或其他特定条件来调整负载均衡策略。

无论使用哪种方法，都需要进行适当的设计和开发工作，以确保多版本共存的正确性和可靠性。此外，需要对多个版本的微服务进行适当的测试，以确保它们在共存的情况下能够正确地处理请求和返回结果。

总结而言，微服务多版本共存是通过合理的请求路由、版本标识和负载均衡策略来进行管理的。选择适合当前需求和系统架构的方法，并经过充分的测试和验证，可以实现平滑的微服务升级和迭代。
